# /prd · 需求调整意图归档闸门

用户每抛一条"我觉得 X 不好/想改 Y"，本技能负责把它从**主观感受**转成
**可归因、可投基础设施的结构化记录**。核心是一道质量闸门:**绝不把模糊感觉
直接写进客观记录里污染数据**。

> 配套事实源:`docs/10_需求调整归因表.md`(类别定义 §一 + 决策表 §二 + 汇总日志 §三)。
> 单条记录落在 `docs/prd-intake/` 文件夹,一条一文件,append-only。

## 触发
```
/prd <一句话感受或诉求>
```
例:`/prd 我觉得流程太卡了`

---

## 流程

### 0 · 读 taxonomy
先读 `docs/10_需求调整归因表.md` §一(N/S/I/O/R 五类)与 §二(类别→基础设施决策表)。
归类与"该投什么基础设施"都以该文件为准,不要凭记忆。

### 1 · 好问题 / 差问题闸门
一条提问要**同时**具备三项才算「好问题(可落盘)」:

| 维度 | 要求 | 反例(差) | 正例(好) |
|---|---|---|---|
| **定位** | 具体到页面/步骤/元素/场景 | "流程" | "工作台出图后选变体那一步" |
| **现象 / 现状缺口** | 痛点:具体卡在什么；新能力:现状为何不够 | "太卡了" | "点生成后 10s 无反馈" ／ "几十张图得逐张设终稿才能导" |
| **期望调整** | 想变成什么样的具体改动/能力 | (没有) | "提交后立刻出现排队占位卡 + 进度" |

> **两类输入,同一套闸门**:痛点反馈(多落 O/S/I)填「现象=卡在哪」;新功能诉求(多落 N)填「现状缺口=现有为何不满足」。别对净新增能力硬凑一个"卡点"。
> **第 4 要素「范围/边界」(条件必填)**:诉求含"全部/所有/批量/某类/历史"等范围词时,必须追问边界(例:"所有出过的图"含不含草稿/改图子版本/仅终稿)。范围歧义会实质改变实现与分权,**N 类尤其必问**。

缺**任意一项** = 差问题 = 模糊感觉 → 进第 2 步追问。
`我觉得流程太卡了` 三项全缺,**必须**追问,不许直接落盘。

### 2 · 追问(可反复)
用 **AskUserQuestion** 把缺的维度问出来,每个维度给候选 chip + 留 Other 自由输入。
**只问缺的那几项**——已具体的维度不要重复问;只有一项有歧义(如 N 类只缺范围)时,单维度追问即可,别为凑形式多问。
若环境没有 AskUserQuestion(非交互场景),退化为纯文本提问,把候选列在问题里。常用候选:
- 定位:列出 5 个页面(首页 / Campaign / 品牌知识库 / 素材库 / 工作台)+ Other。
- 现象/现状缺口:等待无反馈 / 找不到入口 / 步骤太多 / 结果看不到 / 报错不可读 / 现有能力覆盖不到 + Other。
- 范围/边界(含范围词时):仅终稿 / 全部 variants / 含改图子版本 / 含草稿 + Other。
- 期望调整:让用户一句话描述「在 X 处加/改 Y」。

原则:
- **宁可多问一轮,不要把模糊写进记录。** 缺的维度答完仍模糊就再问,直到要素齐(可反复)。
- 若用户明确"先记着、暂时说不清" → 允许以 `quality: draft` 落盘,显式标注「未澄清」,
  §三 行的「质量」列写 `draft`,**不计入** §四 分布统计(避免污染分布)。

### 3 · 归类 + 根因(信息齐后)
- 选**一个主代号** **N/S/I/O/R**(依 §一)。判别要点:
  - 后端/功能在、只是页面上看不到/确认不了 → **O**(可见性·验收缺口)。
  - 需求本来就有歧义/没定义"算完成" → **S**。
  - spec 写清了没做到 → **I**。 范围里真没有 → **N**。 纯改进/重构 → **R**。
  - **N 判别补充**:复用已有代码/底座 **≠** 已有需求——只要产品文档(`docs/02`·`docs/03`)没写过这条能力,即使能在现有 plumbing 上长出来,也归 **N**。先去代码/文档取证再下结论(对齐 CLAUDE.md §0 规则2:Investigate)。
- **叠类规则**:每条**只取一个主代号**用于 §四分布统计;确有次因可在记录与 §三里写 `(叠 S)` 作注,但**次因不计入统计**,避免双算污染分布。
- 写**一句根因**:哪个产物缺位(spec / 契约 / 可见性 / DoD / 回归 / backlog)。
- 对应**基础设施动作**(查 §二 决策表)。
- **目标/下一步**:这条要达成什么、归谁。

### 4 · 落盘(收集完才写,两处)
a. 新建 `docs/prd-intake/<YYYY-MM-DD>-<短slug>.md`(单条事实记录,见下方模板)。
   - **slug 规范**:中文短词,2–6 字概括,连字符无所谓(如 `工作台出图无即时反馈`)。
   - **文件名两处必须逐字一致**:记录文件名 ⇄ §三链接里的文件名,写完回看对账一次。
b. 给 `docs/10_需求调整归因表.md` §三 日志表**追加一行**,行号取**现有最大行号 + 1**(并发/多人编辑时同理,别撞号)。「AI 原话/提议」列链接到记录文件。**行的列骨架**(8 列,与表头对齐):
   ```
   | <max+1> | <YYYY-MM-DD> | [原话](./prd-intake/<文件名>) | **<主代号>** | <一句根因> | <基础设施动作> | good｜draft | 待办 |
   ```
c. commit + push(append-only、低风险,直接提交;subject 形如
   `docs(prd-intake): 归档 <slug>`)。不建 PR。

## 单条记录模板 — `docs/prd-intake/<日期>-<slug>.md`
```markdown
# <一句话标题>

- **日期**:YYYY-MM-DD
- **归类**:<N/S/I/O/R 之一>｜<类别名>   <可选 (叠 X)>
- **质量**:good ｜ draft(未澄清)

## 原始提问
> <用户 /prd 后面的原话,逐字保留>

## 追问澄清(Q&A)
> 好问题(三要素已齐)→ 跳过追问,此段写"已齐故跳过 + 命中判定";差问题→逐条记 Q&A。
- Q(定位):… → A:…
- Q(现象/现状缺口):… → A:…
- Q(范围/边界,含范围词时):… → A:…
- Q(期望调整):… → A:…

## 三要素(澄清后)
- 定位:<页面/步骤/元素/场景>
- 现象/现状缺口:<痛点的卡点 或 新能力的现状不足>
- 范围/边界:<有则写,无则省>
- 期望调整:<在 X 处加/改 Y>

## 归因
- 根因(一句):<哪个产物缺位>
- 基础设施动作:<查 docs/10 §二>
- 目标/下一步:<达成什么·归谁>
```

## 边界
- 这条技能只**记录与归因**,不在本次顺手把功能改了(除非用户另外明确要求)。
- 不裁决"该不该做"——归因不裁决,聚合后看分布(§四 复盘)才决定基础设施投向。
