# /prd · 需求调整意图归档闸门

用户每抛一条"我觉得 X 不好/想改 Y"，本技能负责把它从**主观感受**转成
**可归因、可投基础设施的结构化记录**。核心是一道质量闸门:**绝不把模糊感觉
直接写进客观记录里污染数据**。

> 配套事实源:`docs/10_需求调整归因表.md`(类别定义 §一 + 决策表 §二 + 汇总日志 §三)。
> 单条记录落在 `docs/prd-intake/` 文件夹,一条一文件,append-only。

## 触发
```
/prd <一句话感受或诉求>
```
例:`/prd 我觉得流程太卡了`

---

## 流程

### 0 · 读 taxonomy
先读 `docs/10_需求调整归因表.md` §一(N/S/I/O/R 五类)与 §二(类别→基础设施决策表)。
归类与"该投什么基础设施"都以该文件为准,不要凭记忆。

### 1 · 好问题 / 差问题闸门
一条提问要**同时**具备三项才算「好问题(可落盘)」:

| 维度 | 要求 | 反例(差) | 正例(好) |
|---|---|---|---|
| **定位** | 具体到页面/步骤/元素 | "流程" | "工作台出图后选变体那一步" |
| **现象** | 具体卡在什么 | "太卡了" | "点了生成后 10s 没有任何反馈,以为没点上" |
| **期望调整** | 想变成什么样的具体改动 | (没有) | "提交后立刻出现一个排队中的占位卡 + 进度" |

缺**任意一项** = 差问题 = 模糊感觉 → 进第 2 步追问。
`我觉得流程太卡了` 三项全缺,**必须**追问,不许直接落盘。

### 2 · 追问(可反复)
用 **AskUserQuestion** 一次把缺的维度问全,每个维度给候选 chip + 留 Other 自由输入。常用候选:
- 定位:列出 5 个页面(首页 / Campaign / 品牌知识库 / 素材库 / 工作台)+ Other。
- 现象:等待无反馈 / 找不到入口 / 步骤太多 / 结果看不到 / 报错不可读 + Other。
- 期望调整:让用户一句话描述「在 X 处加/改 Y」。

原则:
- **宁可多问一轮,不要把模糊写进记录。** 答完仍模糊就再问,直到三项齐(可反复)。
- 若用户明确"先记着、暂时说不清" → 允许以 `quality: draft` 落盘,显式标注「未澄清」,
  **不计入** §三 汇总统计(避免污染分布)。

### 3 · 归类 + 根因(信息齐后)
- 选一个代号 **N/S/I/O/R**(依 §一)。判别要点:
  - 后端/功能在、只是页面上看不到/确认不了 → **O**(可见性·验收缺口)。
  - 需求本来就有歧义/没定义"算完成" → **S**。
  - spec 写清了没做到 → **I**。 范围里真没有 → **N**。 纯改进/重构 → **R**。
- 写**一句根因**:哪个产物缺位(spec / 契约 / 可见性 / DoD / 回归)。
- 对应**基础设施动作**(查 §二 决策表)。
- **目标/下一步**:这条要达成什么、归谁。

### 4 · 落盘(收集完才写,两处)
a. 新建 `docs/prd-intake/<YYYY-MM-DD>-<短slug>.md`(单条事实记录,见下方模板)。
b. 给 `docs/10_需求调整归因表.md` §三 日志表**追加一行**,「AI 原话/提议」列用
   `[原话](./prd-intake/<文件名>)` 链接到记录文件。
c. commit + push(append-only、低风险,直接提交;subject 形如
   `docs(prd-intake): 归档 <slug>`)。不建 PR。

## 单条记录模板 — `docs/prd-intake/<日期>-<slug>.md`
```markdown
# <一句话标题>

- **日期**:YYYY-MM-DD
- **归类**:O｜可见性·验收缺口   ← N/S/I/O/R 之一
- **质量**:good ｜ draft(未澄清)

## 原始提问
> <用户 /prd 后面的原话,逐字保留>

## 追问澄清(Q&A)
- Q(定位):… → A:…
- Q(现象):… → A:…
- Q(期望调整):… → A:…

## 三要素(澄清后)
- 定位:<页面/步骤/元素>
- 现象:<具体卡点>
- 期望调整:<在 X 处加/改 Y>

## 归因
- 根因(一句):<哪个产物缺位>
- 基础设施动作:<查 docs/10 §二>
- 目标/下一步:<达成什么·归谁>
```

## 边界
- 这条技能只**记录与归因**,不在本次顺手把功能改了(除非用户另外明确要求)。
- 不裁决"该不该做"——归因不裁决,聚合后看分布(§四 复盘)才决定基础设施投向。
