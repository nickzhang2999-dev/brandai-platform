# 2026 年 7 月 5 日 V0.0.8 修改内容统计

## 版本定义

本次修改定义为 **V0.0.8**，工程包版本同步调整为 `0.0.8`。

## 修改背景

V0.0.7 已支持在 AI 工作台从素材库选择素材，并区分：

- 必须 100% 调用
- 仿制借鉴

但实际生成链路中，「必须 100% 调用」仅被转译成提示词和参数记录，未真正把素材图片作为图像输入交给图片模型，因此无法保证素材一定出现在生成图中。

## 修改内容

### 1. STRICT 调用改为真实图像输入链路

当工作台选择素材并设置为「必须 100% 调用」时：

- 系统不再只走纯文生图 `/images/generations`。
- AI 服务会识别 `ReferenceImage.mode = STRICT`。
- 生成时改走 `provider.edit()` 图像输入链路。
- 被选择的素材图片会作为源图传入 provider。
- prompt 中继续携带项目描述、品牌规则、用户提示词和保真约束。

### 2. 防止静默降级

V0.0.8 明确要求 STRICT 不能静默降级成纯提示词：

- 如果一个请求中出现多张 STRICT 素材，Web API 会直接拒绝。
- AI 服务也会做二次防御校验。
- 生成版本参数会写入 `generationPath: strict_image_input`。
- 生成版本参数会写入 `strictReferencePolicy: provider.edit image input; no text-only fallback`。

### 3. 当前限制

V0.0.8 当前仅支持 **1 张 STRICT 主素材**。

原因：

- 多张 100% 保真素材需要画布合成、位置约束、遮罩或多图编辑接口配合。
- 当前先保证单个 Logo / 产品主图可以通过真实图像输入链路执行。
- 其余参考素材应使用「仿制借鉴」。

### 4. 前端本地状态修复

工作台中切换素材调用方式后，会同步更新本地参考素材暂存盘，减少刷新后从 `STRICT` 回退为 `INSPIRATION` 的情况。

说明：服务端 `ProjectAsset` 关系当前只有 `kind`，尚未持久化每个参考素材的调用模式。跨设备保留调用模式需要后续为服务端关系补充 mode / metadata 字段。

### 5. 测试补充

新增 AI 服务测试：

- STRICT 参考素材必须走 `provider.edit()`。
- STRICT 参考素材不得走 `provider.generate()`。
- 多个 STRICT 参考素材会被拒绝。

## 研发注意事项

- `STRICT` 表示必须走图像输入链路，不能只靠 prompt。
- `INSPIRATION` 仍保留现有的提示词参考逻辑。
- 如果后续要支持多张 STRICT 素材，应进入画布合成 / 多图编辑 / 坐标约束设计，而不是直接放宽数量限制。
