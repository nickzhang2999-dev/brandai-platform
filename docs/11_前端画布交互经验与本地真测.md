# 11 · 前端画布交互经验 + 本地真测 playbook（binding）

> **为什么有这份文档。** 2026-06-27/28 一轮里，工作台开放世界画布连续暴露 **4 个交互 bug**

## A · 四类画布/交互 bug 模式（根因 → 规则）

任何「可点、可拖、可缩放、可编辑」的自定义 DOM 画布（`OpenCanvas.tsx` 及未来同类）改动前，
逐条对照：

### A1 · 画布内的浮层 UI 必须 `onPointerDown` `stopPropagation`
- **症状**：点工具条按钮 → 选择被清空 / 工具条瞬间消失 / 按钮"点了没反应"。
- **根因**：画布容器自己监听 `onPointerDown`（框选 / 取消选中：`if(!shiftKey) setSelected(new Set())`）。
  浮层按钮的 `pointerdown` **先于** `click` 冒泡到容器 → 清空选择 → 浮层在 `click` 真正执行**之前**
  就被卸载。
- **规则**：画布内所有浮层容器（工具条、操作条、缩放条、输入框…）的**外层 `div`** 都加
  `onPointerDown={(e) => e.stopPropagation()}`。光给按钮加不够——按钮间隙、输入框留白也会冒泡。

### A2 · 浅层 URL 同步用 `history.replaceState`，不用 `router.replace`
- **症状**：页面"反复抖动"，DevTools Network 满屏 `/workspace`（或 `?_rsc=`）请求、最后一条 pending。
- **根因**：把"当前项目 / 当前出图"反向同步进 URL 时用了 `router.replace(pathname?query)`。Next App
  Router 对**同路由的 query 变更**会发起 **RSC 软导航** → 组件重渲 + `useSearchParams` 变更 → 多个
  同步 effect 互相激发成**导航环**，画面闪烁。
- **规则**：仅为「刷新/分享落同一状态」而写 URL 时，用
  `window.history.replaceState(null, "", \`${pathname}?${params}\`)`（浅层，不发导航、不重取 RSC）。
  SSR / 刷新 / 深链仍从真实地址栏解析，分享不受影响。`router.replace` 只用于**真正要换页**的跳转。

### A3 · 操作即发 + 禁用整条工具条 = "卡死"；长异步必须 arm→confirm + 有界超时
- **症状**：点一个操作按钮，整条工具条变灰、显示"处理中…"，再也点不动、没有出口 = 用户说的"卡住"。
- **根因**：点 op **立即**发起真实长任务（改图等）并 `disable` 整条工具条；没有「先输指令再确认」
  的步骤，也没有超时出口 → 任务一慢/一卡就锁死。
- **规则**：
  1. **arm → confirm**：点 op 只「选中该操作（高亮）」，输入指令后**回车或点「出图/确认」**才真正发起。
     布局固定不随交互增减元素（否则居中工具条会左右抖）。
  2. **有界中间态（CLAUDE.md §0.3 / §2.4）**：任何客户端轮询的长操作都要有上界超时（默认 6min），
     超界**清状态解锁 + 给可读出口**（"可能仍在后台，稍后查看/重试"），绝不无限转。

### A4 · `setPointerCapture` 吞掉原生 `dblclick`；`autoFocus` 被 in-flight click 立刻 blur
- **症状**：双击画布文字**进不了编辑**；或编辑框"一闪而过"；或编辑完点别处**改动不保存**。
- **根因（三连）**：
  1. 元素拖拽在**容器**上 `setPointerCapture(pointerId)` 后，浏览器把后续 `click`/`dblclick`
     **重定向到容器**，元素自己的 `onDoubleClick` **永不触发**。
  2. 进编辑由 `pointerdown` 触发；`<textarea autoFocus>` **同步聚焦**会被「正在进行的 click 的焦点
     结算」立刻 `blur` → `onBlur` 立即清空编辑态 → 编辑框一闪而过。
  3. 点画布（普通 `div`，不可聚焦）**不会**让 `textarea` 失焦 → `onBlur` 永不触发 → 编辑**提交不掉**。
- **规则**：
  1. **手动双击检测**：在 pointerdown 处理里按时间判定（同元素 350ms 内二次 pointerdown = 双击 →
     进编辑），不要依赖原生 `dblclick`。
  2. **延后一帧聚焦**：去掉 `autoFocus`，编辑态置位后 `setTimeout(() => ref.current?.focus(), 0)`，
     等当前 click 结算完再聚焦。
  3. **点框外主动提交**：编辑期监听 `document` 的 `pointerdown`（capture），按在编辑框之外即主动
     `ref.blur()` 触发 `onBlur` 提交；并支持 `Esc` / `⌘·Ctrl+Enter` 提交。

> 以上四类都已在 `apps/web/src/app/(brandai)/workspace/OpenCanvas.tsx` + `page.tsx` 修复，可作参考实现。

## B · 本地真测 playbook（localhost 绕过 MITM 代理，沙箱内可跑）

### B0 · 为什么必须本地起栈
沙箱的策略代理（MITM）会**关掉 headless Chromium 到外网预览域**（`*.geole.me` 走代理 →
`ERR_CONNECTION_CLOSED`，curl/node 能过、浏览器过不去）。但 **`localhost`/`127.0.0.1` 在 noProxy 名单里**
→ 本地起全栈 + Chromium 直连 `127.0.0.1:3000` 是**唯一**能在沙箱里真点交互、自测 UI 的路径。
（`typecheck/build/L1` 不碰指针/焦点/路由，必须真浏览器补这一层。）

### B1 · 起栈（无 docker，native 二进制；本仓库沙箱已验证可行）
- **postgres**：必须以 **`postgres` 用户**（非 root）跑；data dir 放 **postgres 用户能 traverse 的路径**
  （`/var/lib/postgresql/...`，不要放 `/tmp/claude-*` 这种 700 路径，postgres 用户进不去）。
  `initdb --auth=trust` → `pg_ctl -l /var/lib/postgresql/pg.log start`（log 也要放可写路径）→ 建 `brandai` role + db。
- **redis**：`redis-server --port 6379 --daemonize yes`（root 可直接跑）。
- **AI**：`apps/ai/.venv` 已存在；`IMAGE_PROVIDER=mock VLM_PROVIDER=mock uvicorn app.main:app --port 8000`
  （mock 零 key，出图秒回，足够测交互；取证截图禁用 mock 见 §0.1，但**交互自测**用 mock 合理）。
- **worker + web**：`.env` 指向本地 pg/redis + `AUTH_ALLOW_DEMO=1` + `ADMIN_EMAILS=<你>`；
  `prisma db push`（throwaway 库别用 `migrate dev`，它会交互卡住）→ `db:seed` → `pnpm -F web worker` + `pnpm -F web dev`。
- prisma engine 沙箱内下载受阻时，curl 取 engine 后用 `PRISMA_QUERY_ENGINE_LIBRARY` 指过去。

### B2 · 登录 + 造数据（API 直造，比 UI 稳）
- demo 登录：`GET /api/auth/csrf` → `POST /api/auth/callback/credentials`（email + csrfToken），拿 cookie jar。
- 造 workspace → project → generation（mock 秒到 SUCCEEDED）。
- **关键坑**：`/workspace` 的 wsId 由 `getOrCreateActiveBrand`（**brand cookie**）解析，**不是** URL 里的
  project。多品牌存在时它会落到**别的旧 workspace** → 你造的 gen 404、画布空。**修法**：给浏览器 context
  注入 `brandai-active-brand=<你的 wsId>` cookie（会过成员校验，安全）。

### B3 · dev server 陷阱（本轮真踩，耗了一个多小时）
- **hot-reload 会喂旧模块**：改了代码、`✓ Compiled` 也打了，浏览器**仍可能跑旧逻辑**。验证一个修复
  "没生效"时，先怀疑**在测旧代码**，别急着改实现。
- **确认在测新代码**：① 临时 `console.log` + `page.on('console')` 抓，看新分支是否执行；②「秒起」的
  `WEB UP` 可疑——可能旧进程没死、在answer。验证修复时**杀干净再起**：
  `pkill -9 -f next-server / "next/dist/bin/next" / "web dev"` → 确认端口 refused → `rm -rf apps/web/.next`
  → 重启 → **等日志出现 `Ready in`** 再测。
- **选择器要 scope**：右侧表单也有 `<textarea>`（卖点），别用全局 `page.locator('textarea')` 误测了它；
  画布内元素用 `[data-testid=canvas-item]`（已加）+ `[data-kind=image|shape|text]`，工具条按钮用
  `aria-label`/文案。**加测试锚点（`data-testid`）是低风险、长期收益的事，鼓励加。**

### B4 · 可复用 harness
本轮的逐功能浏览器测试已沉淀为 [`tests/interaction/canvas-functions.mjs`](../tests/interaction/canvas-functions.mjs)
（同目录 `README.md` 是一键起栈 + 跑测的步骤）。下次改画布，**先把它跑绿再交付**，别再从零写。

## C · 一句话沉淀

> **编译绿 ≠ 能用。** 指针冒泡、焦点结算、`setPointerCapture`、RSC 软导航、`router.replace`——这些
> 只有**真浏览器真点**才暴露。交互类 UI 的「完成」判据是 **localhost 上 Chromium 逐功能点过且断言通过**，
> 不是 `typecheck/build/L1` 三绿。
