# BrandAI｜需求调整归因表 v0.2

> **本表的本质是归因，不是问题清单。** 把编码 AI 每次的"需求调整提议"归到一个固定类别，**聚合后看分布**；分布直接告诉我们该把基础设施的钱投在哪。
>
> 两个原则：
> 1. **不冤枉 AI** —— 责任列把"需求没写清"和"AI 没做到"分开。
> 2. **不拒绝提议** —— 每一类都有去向（backlog / DoD 前置 / CI 回归 / CDS 预览 / 技术债清单），而不是简单的 yes/no。
>
> 与 `README.md` 顶部进度表互补：进度表记「做没做」，本表记「提议从哪来」。

---

## 一、六个类别（N / S / I / O / R / E）

| 代号 | 类别 | 一句话 | 不冤枉谁 | 根在哪个产物 |
|---|---|---|---|---|
| **N** | 新增需求 | 原始范围里真没有 | 谁都不怪，是新范围 | 产品 backlog |
| **S** | 需求没想清楚 | 需求在，但留了歧义/缺口 | 不怪 AI，怪 spec | `docs/` PRD、字段文档 |
| **I** | 没按需求做 | spec 写清了，AI 没交付 | 这才是 AI 的实现缺口 | 代码 / 契约 |
| **O** | **可见性·验收缺口** | **做了，但验收方在页面上看不到/确认不了** | **不怪 AI，怪没有验收手段** | **CDS 预览 + 视觉测试** |
| **R** | 优化·重构提议 | 既非 bug 也非范围，AI 主动建议改进 | 中性，是判断题 | 技术债清单 |
| **E** | **环境·部署·运行时·验收方法** | **代码/需求都对，但线上跑旧 commit / 缓存盖住 / worker 未就绪 / 测错环境** | **不怪 AI 也不怪测试者，怪上线判据缺失** | **部署流水线 / 运行时配置 / `cds-deploy-verify`** |

> **为什么从 5 类扩到 6 类（v0.2 加 E）**：用真实测试者反馈压测时发现，最常见的争议是「这到底是 bug 还是没部署/缓存？」——代码取证显示代码正确、仅线上复现（典型如线上创建失效）。这既不是 I（代码没做到），也不是 O（做了但看不清）——**O 的隐含前提是代码已正确上线**。它属于运行时/部署/验收姿势，N/S/I/O/R 全装不下；硬塞进 I 就**冤枉 AI**、污染分布。这正是本仓库的头号失败模式（CLAUDE.md §0 规则4「部署成功 ≠ 新代码在跑」、`cds-deploy-verify` 技能都在防它）。
>
> **判 E 的纪律**：E 不能靠静态代码盖棺——必须**运行时取证**（`/api/health` + `.next/CDS_DEPLOY_COMMIT` 比对 HEAD + 一个反映新行为的端点冒烟，见 `cds-deploy-verify`）。确证前先标「疑似 E」，证实是部署/缓存才归 E，证实是代码缺陷才归 I。

## 二、类别 → 该投的基础设施（决策表）

提议集中在哪一类，就投哪一格：

| 类别集中 | 说明 | 该投的基础设施 | 降验收成本? |
|---|---|---|---|
| **N 多** | AI 在帮你发现范围 | 把 backlog 流程跑顺即可，不是 infra 问题 | 否（但别拒绝） |
| **S 多** | spec 模板缺"完成定义" | **验收标准（DoD）和需求同时写** —— 前置，不是"做完再补" | 是，前置最省 |
| **I 多** | 回归/交付没卡住 | 契约测试 + 视觉回归 + CDS 冒烟，在合并前抓 | 是 |
| **O 多** | 功能在但不可见 | **CDS 云预览 + 视觉化自动测试**（主投资线） | 是，最直接 |
| **R 多** | AI 想重构 | 单独技术债清单，批量审，别混进验收 | 中性 |
| **E 多** | "是 bug 还是没部署"反复扯皮 | **上线判据自动化**：部署后 health + `CDS_DEPLOY_COMMIT` 指纹比对 + 新行为冒烟（`cds-deploy-verify`）；缓存破除 | 是，省下扯皮成本 |

## 四、复盘节奏

每隔一批提议看一次分布：

- **O / S 占多** → 加码 CDS 预览 + 视觉测试 + DoD 模板（主投资线被验证）。
- **I 占多** → 加码 CI 回归（契约 + 视觉回归）。
- **N 占多** → 是好事，顺 backlog 即可。
- **R 占多** → 收进技术债清单批量审。
- **E 占多** → 加码上线判据自动化（`cds-deploy-verify` 进 CI/部署后必跑）+ 缓存破除；同时校准验收方的测试姿势（确认测的是新 commit）。

> **集合型吐槽**：一条输入若是"问题很多/到处都是/根本没法用"（如 `连基本测试都过不了`），它不是一条需求而是**多条的集合**。`/prd` 技能的「拆解闸门」会先拆成多条、各跑三要素、各落一行；拆不动的残留部分落 `draft` 不计入分布。**严禁把集合硬塞成一行**（会把不同代号混算，污染统计）。
