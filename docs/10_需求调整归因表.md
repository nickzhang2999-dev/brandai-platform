# BrandAI｜需求调整归因表 v0.1

> **本表的本质是归因，不是问题清单。** 把编码 AI 每次的"需求调整提议"归到一个固定类别，**聚合后看分布**；分布直接告诉我们该把基础设施的钱投在哪。
>
> 两个原则：
> 1. **不冤枉 AI** —— 责任列把"需求没写清"和"AI 没做到"分开。
> 2. **不拒绝提议** —— 每一类都有去向（backlog / DoD 前置 / CI 回归 / CDS 预览 / 技术债清单），而不是简单的 yes/no。
>
> 与 `README.md` 顶部进度表互补：进度表记「做没做」，本表记「提议从哪来」。

---

## 一、五个类别（N / S / I / O / R）

| 代号 | 类别 | 一句话 | 不冤枉谁 | 根在哪个产物 |
|---|---|---|---|---|
| **N** | 新增需求 | 原始范围里真没有 | 谁都不怪，是新范围 | 产品 backlog |
| **S** | 需求没想清楚 | 需求在，但留了歧义/缺口 | 不怪 AI，怪 spec | `docs/` PRD、字段文档 |
| **I** | 没按需求做 | spec 写清了，AI 没交付 | 这才是 AI 的实现缺口 | 代码 / 契约 |
| **O** | **可见性·验收缺口** | **做了，但验收方在页面上看不到/确认不了** | **不怪 AI，怪没有验收手段** | **CDS 预览 + 视觉测试** |
| **R** | 优化·重构提议 | 既非 bug 也非范围，AI 主动建议改进 | 中性，是判断题 | 技术债清单 |

> **为什么把原来的 3 类（新增/没想清楚/没做完）扩成 5 类**：很多提议是"功能在、但看不清楚"（O）或"做了、但验收没法确认"。它们既不是 I（没做），也不是 N（新需求）。把 O 单列出来，正好对上"CDS 预览 + 视觉化自动测试降验收成本"这条主投资线。

## 二、类别 → 该投的基础设施（决策表）

提议集中在哪一类，就投哪一格：

| 类别集中 | 说明 | 该投的基础设施 | 降验收成本? |
|---|---|---|---|
| **N 多** | AI 在帮你发现范围 | 把 backlog 流程跑顺即可，不是 infra 问题 | 否（但别拒绝） |
| **S 多** | spec 模板缺"完成定义" | **验收标准（DoD）和需求同时写** —— 前置，不是"做完再补" | 是，前置最省 |
| **I 多** | 回归/交付没卡住 | 契约测试 + 视觉回归 + CDS 冒烟，在合并前抓 | 是 |
| **O 多** | 功能在但不可见 | **CDS 云预览 + 视觉化自动测试**（主投资线） | 是，最直接 |
| **R 多** | AI 想重构 | 单独技术债清单，批量审，别混进验收 | 中性 |

## 四、复盘节奏

每隔一批提议看一次分布：

- **O / S 占多** → 加码 CDS 预览 + 视觉测试 + DoD 模板（主投资线被验证）。
- **I 占多** → 加码 CI 回归（契约 + 视觉回归）。
- **N 占多** → 是好事，顺 backlog 即可。
- **R 占多** → 收进技术债清单批量审。
